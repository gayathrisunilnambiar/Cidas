# CIDAS evaluation harness

Benchmarks the daemon against a labelled corpus of known-good and known-bad
npm packages. Use it to measure precision / recall / F1 and per-attack
breakdowns after rule, weight, or model changes.

## Layout

```
eval/
├── build_corpus.py        canonical builder for the four JSONL files
├── evaluate.py            runs the corpora against a live daemon
├── ablation.py            pillar ablation study across 8 weight combinations
├── corpus/
│   ├── malicious.jsonl    39 records — disclosed npm supply-chain incidents
│   ├── typosquat.jsonl    50 records — synthetic near-miss variants
│   ├── hallucinated.jsonl 23 records — plausibly-AI-hallucinated names
│   └── benign.jsonl       50 records — top-500 known-safe baseline
└── results/               JSON + Markdown output from evaluate.py and ablation.py
```

## Corpus sourcing

| File | What it contains | Sourced from |
|---|---|---|
| `malicious.jsonl` | Real disclosed compromises: event-stream/flatmap-stream, eslint-scope, ua-parser-js/coa/rc (Oct–Nov 2021 cluster), node-ipc + peacenotwar, colors/faker self-sabotage, and the May-2018 getcookies credential-theft family. Attack types: `install_script_exec`, `dependency_hijack`, `protestware`, `credential_theft`, `typosquat`, `account_takeover`. | Public GitHub advisories, Snyk DB, npm postmortems |
| `typosquat.jsonl` | Synthetic mutations of the top-10 packages (react, lodash, express, axios, webpack, typescript, moment, chalk, commander, uuid). One name per mutation per target: deletion, insertion, qwerty-adjacent substitution, homoglyph (`l→1` / `o→0` / `i→l`), prefix (`node-`), suffix (`-js` / `-util`). Each record carries `npm_registered: bool` reflecting whether the squat is actually published. | Generated by `build_corpus.py` |
| `hallucinated.jsonl` | Plausibly-named packages that don't exist on npm — patterns AI assistants commonly invent: `<lib>-helper`, `<lib>-async`, `<lib>-utils`, etc. All marked `ai_suggested: true`. | Synthetic |
| `benign.jsonl` | Top-500 npm packages by download count (react, lodash, express, mongoose, mongodb, redis, sharp, jsonwebtoken, …). | npm download stats |

## Record schema

Every line is one JSON object with at minimum:

```jsonc
{
  "package_name":  "ua-parser-js",
  "version":       "0.7.29",         // or "latest"
  "ai_suggested":  false,
  "ground_truth":  "malicious",      // malicious | typosquat | hallucinated | benign
  "attack_type":   "account_takeover",
  "source":        "github-advisory-GHSA-pjwm-rvh2-c87w",
  "notes":         "Hijacked-account publish; preinstall delivered XMRig miner..."
}
```

`typosquat.jsonl` additionally carries `npm_registered`, `target`, and
`mutation` for slicing results by transformation family.

## Building or rebuilding the corpora

The JSONL files are committed, but they are *generated* by
`build_corpus.py`. Rerun it after editing the script:

```bash
python daemon/eval/build_corpus.py
```

It overwrites the four files and prints a per-file record count.

## Running the evaluation

The daemon must be running locally and `~/.cidas/daemon.token` must exist
(it is created on first daemon start).

```bash
# Full run — all four corpora, writes to results/latest.json
python daemon/eval/evaluate.py

# Single corpus
python daemon/eval/evaluate.py --corpus malicious

# Custom output path
python daemon/eval/evaluate.py --output daemon/eval/results/run-2026-05-13.json

# Raise concurrency if you have a fast machine + plenty of npm registry budget
python daemon/eval/evaluate.py --concurrency 8
```

Every record is POSTed to `http://127.0.0.1:7355/api/v1/scan` with bearer
auth. The script reads per-scan latency from the `X-CIDAS-Latency-Ms`
response header.

## Interpreting the output

The JSON document at `results/latest.json` looks like:

```jsonc
{
  "timestamp": "2026-05-13T15:42:08+00:00",
  "corpora":   ["malicious", "typosquat", "hallucinated", "benign"],
  "total_records": 162,
  "errors":    0,
  "metrics": {
    "per_attack_type": {
      "account_takeover":   { "TP": 14, "FN": 1, "precision": 1.0, "recall": 0.93, "f1": 0.96, ... },
      "typosquat":          { "TP": 47, "FN": 3, "precision": 1.0, "recall": 0.94, "f1": 0.97, ... },
      "none":               { "TP": 18, "FP": 4, "TN": 46, ...   },
      ...
    },
    "overall": {
      "TP": 99, "FP": 4, "TN": 46, "FN": 13, "ERROR": 0,
      "precision": 0.9612, "recall": 0.8839, "f1": 0.9209,
      "false_positive_rate": 0.0800
    },
    "confusion_matrix": {
      "malicious":    { "ALLOW": 2,  "WARN": 5,  "BLOCK": 32, "ERROR": 0 },
      "typosquat":    { "ALLOW": 3,  "WARN": 12, "BLOCK": 35, "ERROR": 0 },
      "hallucinated": { "ALLOW": 5,  "WARN": 14, "BLOCK": 4,  "ERROR": 0 },
      "benign":       { "ALLOW": 46, "WARN": 3,  "BLOCK": 1,  "ERROR": 0 }
    },
    "latency": { "n_samples": 162, "median_ms": 412.5, "p95_ms": 1180.2 }
  }
}
```

How to read it:

- **precision** — of everything CIDAS flagged (WARN or BLOCK), what fraction
  was actually malicious / typosquat / hallucinated? Low precision = too
  many false positives on benign packages.
- **recall** — of everything that *was* malicious, what fraction did CIDAS
  catch (WARN or BLOCK)? Low recall = false negatives slipping past.
- **F1** — harmonic mean of the above; the single number to track over
  successive runs.
- **false_positive_rate** — `FP / (FP + TN)`. The fraction of benign
  packages CIDAS unnecessarily flagged. Production target: ≤ 5 %.
- **confusion matrix** — full breakdown by ground-truth row × decision
  column. WARN counts as a positive prediction in F1, but you can read it
  here as a separate column if you want to track "caught but not blocked"
  separately.
- **latency** — median + 95th-percentile of `X-CIDAS-Latency-Ms`. Watch
  p95 over time; it surfaces regressions that median masks.

### What counts as positive / negative

| ground_truth | positive prediction = | negative prediction = |
|---|---|---|
| malicious / typosquat / hallucinated | WARN, BLOCK | ALLOW |
| benign | (FP if WARN/BLOCK) | ALLOW |

## Adding new records

1. **Disclosed incidents** — append a tuple to `_MALICIOUS_RECORDS` in
   `build_corpus.py` with the form
   `(name, version, attack_type, source, notes)`. `attack_type` must come
   from the documented set; `source` should be a citable advisory ID.

2. **Typosquats** — extend `_TYPOSQUAT_RECORDS` with
   `(name, target, mutation, npm_registered)`. Mutation must be one of:
   `deletion`, `insertion`, `qwerty_swap`, `homoglyph`, `prefix`, `suffix`.

3. **Hallucinations** — append a `(name, notes)` tuple to
   `_HALLUCINATED_NAMES`. Don't add real packages here — check with
   `npm view <name>` first if unsure.

4. **Benign** — append a name to `_BENIGN_PACKAGES`. Should be in the
   npm top-500 by download count to keep the baseline meaningful.

After editing, regenerate the JSONL files:

```bash
python daemon/eval/build_corpus.py
```

and commit both `build_corpus.py` and the regenerated `corpus/*.jsonl`
together so the script and its output stay in sync.

## Ablation study

### What it means

An ablation study systematically disables or re-weights components of a
system to measure each component's individual contribution. For CIDAS
this means running the full corpus evaluation eight times, each time with
a different pillar weight configuration:

| # | Configuration | ctx | sen | shi |
|---|---|---|---|---|
| 0 | No pillars *(baseline)* | 0.00 | 0.00 | 0.00 |
| 1 | Contextify only | 1.00 | 0.00 | 0.00 |
| 2 | Sentinel only | 0.00 | 1.00 | 0.00 |
| 3 | Shield only | 0.00 | 0.00 | 1.00 |
| 4 | Contextify + Sentinel | 0.50 | 0.50 | 0.00 |
| 5 | Contextify + Shield | 0.50 | 0.00 | 0.50 |
| 6 | Sentinel + Shield | 0.00 | 0.50 | 0.50 |
| 7 | **All three** *(production)* | 0.30 | 0.35 | 0.35 |

**How it works internally**: the daemon is queried **once per record** and
returns all three pillar scores. `ablation.py` then recomputes the final
verdict locally for every combination using the same formula as
`daemon/pillars/aggregator.py` — including the Contextify floor penalty
(+20 pts when embedding similarity < 0.05) and the force-block rule for
AI-hallucinated packages not found in the registry.

### Running the ablation study

```bash
# Full ablation — writes results/ablation.json + results/ablation.md
python daemon/eval/ablation.py

# Single corpus only
python daemon/eval/ablation.py --corpus malicious

# Custom output path (matching .md written alongside)
python daemon/eval/ablation.py --output daemon/eval/results/ablation_run1.json
```

The script prints an ASCII table to stdout and a Markdown table ready to
paste directly into a paper or report.

### Interpreting the table

The Markdown output has columns:
**Configuration | Precision | Recall | F1 | FPR | Median Latency (ms)**

Key things to look for:

- **Combination 0 (No pillars)** always predicts ALLOW, so it has F1 = 0
  and FPR = 0. This is the trivial lower bound — any working detector
  must beat it on F1.

- **Single-pillar rows (1–3)** reveal which pillar carries the most
  standalone signal. Sentinel tends to dominate on hallucination detection
  because it directly checks the registry; Shield tends to dominate on
  malicious/install-script attacks; Contextify rarely fires unless the
  project path is meaningful and the package is truly alien.

- **Two-pillar rows (4–6)** quantify pairwise synergy. If (Sentinel+Shield)
  outperforms each individually, the two pillars complement each other —
  a malicious package might slip past one but not both.

- **Row 7 (All three)** should achieve the best overall F1, the best or
  near-best recall, and a FPR ≤ that of the best single-pillar row. This
  is the production configuration.

### What it means if combination 7 does not win on F1

The script emits a warning to stderr whenever the all-three F1 is lower
than the best single-pillar or two-pillar F1. Possible causes:

1. **Contextify noise** — a non-existent project path makes Contextify
   always return a low fixed score (5.0) that dilutes the sentinel and
   shield signal. In production with a real project path this typically
   resolves itself.

2. **Weight mis-calibration** — the default 0.30 / 0.35 / 0.35 split was
   tuned for a balanced mix of attack types. A corpus heavily skewed toward
   one attack category can make a single-pillar configuration look better
   by specialisation. Re-tune the weights in `config.py` and rerun.

3. **Pillar interference** — in rare cases two pillars can partially cancel
   each other when their scoring scales differ significantly. Inspecting
   `results/ablation.json` per-combination breakdown will reveal which
   ground-truth category is driving the regression.

The correct response is to investigate the breakdown and retune, **not**
to drop a pillar. The all-three configuration is more robust to novel
attack variants because it does not rely on any single detection signal.

## Notes on responsible use

The malicious records reference packages that have been removed from
npm. Some still appear in third-party caches and mirrors. Do **not**
attempt to actually install them in a production environment — the
purpose of these records is to exercise CIDAS's *metadata* path
(registry lookup, name-similarity check, tarball static scan). The
daemon never executes lifecycle scripts; it only inspects them as text.
