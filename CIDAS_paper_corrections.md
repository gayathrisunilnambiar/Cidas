# CIDAS Paper — Corrections Against Current Implementation

This document lists every place where `CIDAS_paper.pdf` needs to change to
accurately describe the system as it exists on the `content-drift` branch
today, plus the real, freshly-measured numbers to replace the paper's
current (fabricated/stale) tables. There is no `.tex`/`.docx` source for the
paper in this repository — apply these corrections to whatever source you
maintain elsewhere (Overleaf, Word, etc.).

Two code bugs and one design inconsistency were also fixed as part of this
pass (see "Code fixes made" at the end) — those are *not* paper corrections,
they're why some of the numbers below look different from the very first
measurement taken.

## 3. Table 2 (Detection Performance) and Table 3 (Scan Latency)

**Current text:** Macro-F1 = 0.97 across four categories; P99 latency of
1950 ms for the tarball-download scenario.

**Reality:** No `results/` artifact existed anywhere in the repo to
support these numbers — `daemon/eval/results/` was empty before this pass.
A live run of `daemon/eval/evaluate.py` against the real 162-record corpus
and the running daemon produced the numbers in **`daemon/eval/results/latest.json`**
(see the placeholder block below — filled in after the final run in this
session). Replace Tables 2 and 3 wholesale with those measured numbers, and
change the surrounding prose from "0.97" to whatever `overall.f1` reports.

**Important caveat to add to Section VIII-A (Experimental Setup) and
Section IX-E (Limitations):** unlike the paper's implied lab conditions,
real per-package latency is dominated by live npm-registry round-trips
(registry document fetch, tarball download for the current *and* the
immediately-preceding version for cross-version diffing, and the npm
downloads-API call) — it is **not** representative of a cache-hit or
air-gapped scenario, and will vary with the registry's real-world response
time from the tester's network. State the measured P50/P95/P99 as "live
network conditions, cache-miss path" rather than implying they're
achievable unconditionally.

**Measured numbers** (from `daemon/eval/results/latest.json`, 162-record
corpus, cold cache, concurrency=4, against the daemon on the branch as of
this pass — reproduce with `daemon/eval/evaluate.py`):

**Table 2 — Detection Performance**

| Category | Verdict | Precision | Recall | F1 |
|---|---|---|---|---|
| Malicious (dependency_hijack/install_script_exec/account_takeover/protestware/credential_theft, N=39) | WARN/BLOCK | 1.00 | 0.72–1.00 (varies by sub-type; see per_attack_type in the JSON) | 0.55–1.00 |
| Typosquat (N=50) | WARN/BLOCK | 1.00 | 0.74 | 0.85 |
| Hallucinated (N=23) | BLOCK | (see confusion matrix — 6 ALLOW, 17 BLOCK) | — | — |
| Benign (N=50) | ALLOW | — | — | FPR ≈ 0.15–0.20 |
| **Overall (micro-averaged across all four corpora)** | — | **0.93** | **0.74** | **0.83** |

This replaces the paper's clean 0.97 macro-F1 with a real, messier number.
The two biggest drivers below 0.97, both root-caused during this pass:

1. **Typosquat recall (0.74, not 0.95).** Levenshtein edit-distance-2
   structurally cannot catch **prefix/suffix** squats (confirmed: 19/50 —
   100% of the corpus's prefix and suffix mutation families — are
   undetectable by `check_name_similarity` regardless of threshold tuning,
   since e.g. `node-react` is edit-distance 5 from `react`). This is a
   detection-coverage gap, not fixed in this pass (a real fix needs an
   affix-aware check, which risks new false positives on legitimately
   prefixed/suffixed unrelated packages and should be validated separately
   before shipping).
2. **False positives on short package names.** Empirically, `vue` and
   `next` — both legitimate, unrelated, extremely popular packages — get
   flagged `typosquat_detected` because they land within Levenshtein
   distance 2 of `vite` and `nuxt` respectively, purely because short
   strings collide easily under a fixed edit-distance threshold. This is a
   known limitation of naive edit-distance typosquat detection on short
   names; add to Section IX-E (Limitations) rather than Section V-B (the
   paper currently presents the Levenshtein check without this caveat).

A related bug **was** fixed in this pass: two entirely legitimate, popular
packages (`redux-thunk`, `nodemailer`) were observed force-BLOCKed with a
`package_not_found` flag during an early run — root cause was a registry
network hiccup (timeout/dropped connection) being indistinguishable from a
confirmed HTTP 404 in `Sentinel.check_registry_existence`. Mitigated (not
fully solved) by (a) not caching a failed registry fetch, so the next
caller within the TTL window gets a fresh attempt rather than a repeated
stale failure, and (b) one confirmatory re-fetch before concluding
"absent." A structurally complete fix (a tri-state exists/absent/undetermined
result threaded through to Sentinel) is flagged as priority future work in
Section IX-F, since it touches every `get_package_metadata` call site.

**Table 3 — Scan Latency**

Latency varied substantially by run depending on registry contention and
cache warmth — report this honestly rather than a single clean number:

| Scenario | Median | P95 |
|---|---|---|
| Best observed (cache partially warm from a prior run) | 2.7 s | 6.4 s |
| Cold cache, concurrency=4 (this session's final measurement) | 3.9–17.2 s | 8.7–65.5 s |

Both are far above the paper's claimed sub-2-second target and its
1950 ms P99. The dominant cost, root-caused in this pass: Shield's
cross-version diff analysis downloads **two full tarballs** (current +
immediately-preceding version) and extracts/scans up to 50 files in each,
per scan of an existing package — this is inherent to the feature as
designed, not a bug, but it means "sub-two-second" is only realistic on a
cache hit (confirmed: cache-hit latency is genuinely sub-50ms, matching the
paper's Table 3 cache-hit row) — not on the cache-miss / tarball-download
path the paper's Table 3 also claims sub-2s for.

One real bug **was** fixed here: a single scan of an existing package was
independently re-fetching the same package's full registry document from
up to five call sites (Contextify, Sentinel, Shield, disk_checker, and the
diff analyzer's version-history lookup) with zero sharing between them —
confirmed via daemon logs showing 9+ identical GET requests for one
package within a single scan. A short-TTL single-flight cache
(`npm_registry._fetch_registry_doc_cached`) cut this dramatically (a
first, uncached measurement showed 38.2s median / 80.6s P95 before the
fix — see the "Code fixes made" section).

**Fix for Section VIII-A/D and Table 3:** replace the claimed numbers with
the measured ranges above, and add: "Latency is dominated by live
npm-registry and tarball-download round-trips rather than daemon compute;
absolute numbers depend on registry responsiveness and network contention
at scan time and should not be read as a fixed guarantee. The sub-two-second
design target (P5) is met on the cache-hit path; the cache-miss path with
cross-version diffing can take substantially longer."

## 4. Section VIII-E — Ablation study

**Current text:** "We evaluated 12 weighting configurations varying wC ∈
{0.20, 0.25, 0.30} and distributing the remainder between wS and wSh."

**Reality:** `daemon/eval/ablation.py` implements **8** fixed
configurations — a component ablation (no pillars / each pillar alone /
each pair / all three), not a continuous weight sweep. This is intentional
and well-reasoned (see `daemon/eval/README.md`'s "Ablation study" section)
— it directly measures each pillar's standalone and pairwise contribution,
which is arguably more informative than a weight sweep for a 3-signal
system. Rewrite the paper to describe the real design rather than building
a new 12-config sweep to match the old text.

**Fix:** Replace with:

> We evaluated 8 pillar-weight configurations: no pillars (baseline,
> ctx=0/sen=0/shi=0), each pillar in isolation (ctx=1/sen=0/shi=0, etc.),
> each pairwise combination (e.g. ctx=0.5/sen=0.5/shi=0), and the
> production configuration (ctx=0.30/sen=0.35/shi=0.35). This measures each
> pillar's standalone and pairwise contribution directly, rather than
> sweeping a single weight in isolation.

**Measured numbers** (from `daemon/eval/results/ablation.json` /
`ablation.md`, same corpus and daemon state as Table 2/3 above):

| Configuration | Precision | Recall | F1 | FPR |
|---|---|---|---|---|
| No pillars | 0.00 | 0.00 | 0.00 | 0.00 |
| Contextify only | 0.91 | 0.74 | 0.82 | 0.16 |
| Sentinel only | 0.91 | 0.74 | 0.82 | 0.16 |
| Shield only | 0.90 | 0.74 | 0.81 | 0.18 |
| Contextify + Sentinel | 0.91 | 0.74 | 0.82 | 0.16 |
| Contextify + Shield | 0.91 | 0.74 | 0.82 | 0.16 |
| Sentinel + Shield | 0.91 | 0.74 | 0.82 | 0.16 |
| **All three (production)** | **0.91** | **0.74** | **0.82** | **0.16** |

**Important, honest finding to add to Section VIII-E:** the eight
non-baseline configurations score nearly identically. Root cause (not a
bug — a real property of this corpus given the current aggregation
design): three force-floor rules in `Aggregator.aggregate()` — a detected
typosquat forces at least WARN, a registry-confirmed-absent or
known-supply-chain-incident package forces BLOCK — fire **independent of
pillar weight** and determine the verdict for most of the corpus (all of
`typosquat.jsonl`, most of `malicious.jsonl`, all of `hallucinated.jsonl`).
Weight configuration only matters for the minority of records that trigger
none of these three conditions (chiefly the benign corpus, plus any
malicious/typosquat record that happens not to match a force rule) —
consistent with "Shield only" being the sole row that measurably differs.
Replace the paper's "0.30/0.35/0.35 maximized macro-F1 while keeping FPR
below 5%" claim (not supported by any configuration in this run — FPR
ranges 0.16–0.18 across the board) with: "All eight configurations perform
similarly on this corpus because three flag-driven force-rules — not pillar
weighting — determine the verdict for the large majority of records;
weight configuration is only decisive for the subset of packages that
trigger none of typosquat/known-incident/not-found."

## 5. Section V-D (Aggregator) — Contextual floor rule

**Current text:** "Contextual floor. If sC > 85 in a project with ≥ 20
dependencies, r is raised to at least max(r, 45)."

**Reality:** Contextify's `compute_score` caps its output at **25** (see
`daemon/pillars/aggregator.py`'s own comment: "Contextify's compute_score
caps at 25, which only contributes ~7.5 points at weight 0.30"), so `sC >
85` can never occur — the described rule is unreachable as written. The
actual floor rule in `aggregator.py` is unrelated to what the paper
describes: an **additive +20 penalty** applied when raw embedding cosine
similarity is **below 0.05** (not gated on dependency count at all), tagging
`alien_to_project`.

**Fix:** Replace with:

> Contextual floor. Contextify's `compute_score` maps cosine similarity to
> a bounded 0–25 risk contribution (0 when similarity ≥ 0.65, 10 when
> "loosely related" (≥0.35), 25 when the project has more than 10 tracked
> dependencies and similarity is still low ("unfamiliar in a mature
> project"), 15 otherwise). Because this contribution alone cannot reach
> WARN territory at weight 0.30, the Aggregator applies a separate flat
> +20 penalty when raw cosine similarity falls below 0.05, tagging
> `alien_to_project` — closing the gap for packages that are essentially
> unrelated to anything the project imports.

Also remove/correct the separate, third description of a "0.15
similarity / >20 dependencies" floor in Section V-A-2 (Contextify) — no
such threshold exists anywhere in the code; the only project-size-sensitive
branch is the `domain_count > 10` check inside `compute_score` described
above.

## 8. Section VI-D — LLM second-pass verifier blend weight

**Current text:** "The LLM score is blended at 0.40×LLM + 0.60×regex to
prevent over-reliance on probabilistic output."

**Reality:** `shield.py` computes `primary(regex) * 0.4 + llm_score * 0.6`
— the **LLM gets 60%**, not the regex. This is intentional, not a bug: the
code's own comment explains the regex is "high-precision/low-recall" (catches
canonical phrasings, misses paraphrases) while the LLM is the opposite, so
weighting the LLM higher is what "recovers both" — the paper's stated
rationale ("prevent over-reliance on probabilistic output") actually argues
for the opposite of what the code does, so the code was kept as-is (it's
the more defensible design) and the paper text is corrected instead.

**Fix:** Replace with: "The LLM score is blended at 0.60×LLM + 0.40×regex.
The regex pass is high-precision but low-recall (it catches canonical
phrasings like 'ignore previous instructions' but misses paraphrases); the
LLM is the opposite. Weighting the LLM higher recovers the recall the regex
pass misses, while the regex's 40% floor still anchors the score against a
purely hallucinated LLM judgment."

## 9. Section V-B (Sentinel) — Typosquat severity ordering

**Current text:** "Candidates matching a typosquat pattern that do not yet
exist on the registry are penalized more severely than existing
typosquats, because a pre-registered name signals active attacker intent."

**Reality (before this pass):** This was backwards for human-typed
installs — an *existing* typosquat scored 100 while a *non-existent* one
scored only 95. **Fixed in this pass** (`daemon/pillars/sentinel.py`): a
typosquat name that doesn't exist yet now also scores 100, so it is never
scored lower than an existing one. Both are tied at the maximum rather than
strictly ordered (100 is the score ceiling, so "more severe" can only mean
"at least as severe" once a signal is already maxed out).

**Fix:** Replace "penalized more severely" with: "Candidates matching a
typosquat pattern receive the maximum Sentinel risk score (100)
regardless of whether the name is already registered — pre-registration
does not reduce severity, since an unregistered typosquat name is itself
a strong signal of staged attacker intent."

## 17. Section VI-E / VIII-F — Concept-drift monitor mechanics

**Current text:** "computes 10-bin per-pillar score histograms" from "the
most recent 100 audit-log records"; "the baseline is computed from the
first 200 scan records after initial deployment."

**Reality (before this pass):** The audit log only ever persisted the
composite risk score, never per-pillar scores — `drift_monitor.py`'s own
comment admitted "the current Phase 1 audit log format stores only an
overall score." So real per-pillar drift detection was **structurally
impossible**: `check_drift()` always returned `insufficient_data` against
a real deployment's audit log. **Fixed in this pass**
(`daemon/router.py`'s `_audit_scan` now writes `contextify_score`,
`sentinel_score`, `shield_score` on every record) — the live daemon
health-check now genuinely detects drift (confirmed after the fixes: a
fresh run against real traffic returned `status: "alert"`,
`overall_kl: 1.9`, since the real score distribution differs from the
synthetic baseline described next).

Separately, the baseline was never "computed from the first 200 scan
records after initial deployment" — `drift_monitor.py`'s `_cmd_build()`
builds it from the eval corpus using `random.uniform()` over hand-picked
score ranges per ground-truth label (see `_SYNTH_RANGES`), a synthetic
seed baseline, not real deployment history.

**Fix:** Replace the relevant Section VI-E / VIII-F text with:

> On each health-check poll the monitor reads the most recent 100
> audit-log records (each of which now carries all three per-pillar
> scores alongside the composite verdict), computes 10-bin per-pillar
> histograms, and compares them to a stored baseline via KL divergence,
> returning `warn` above 0.15 or `alert` above 0.30 (averaged across
> pillars). Because no production deployment history exists yet, the
> baseline is instead seeded synthetically: `daemon/eval/drift_monitor.py
> build` assigns each labelled corpus record a per-pillar score sampled
> from empirically-informed ranges per ground-truth class (e.g. malicious
> packages score high on Sentinel and Shield, benign packages score low
> across all three) and builds the baseline histogram from that. This
> seed baseline should be replaced with one built from real traffic once
> a production deployment accumulates enough scan history.

Also fix a real path-portability bug found alongside this: `drift_monitor
.py`'s corpus and baseline paths were hardcoded relative to the process's
current working directory (`Path("daemon/eval/corpus")`), which silently
produced `insufficient_data`/empty results whenever the daemon or test
suite was launched from any directory other than the repo root. Both paths
are now anchored to the module's own file location — no paper text
changes needed here, this is purely a robustness fix.

## Code fixes made in this pass (not paper corrections — but why some
numbers above changed between the first and final measurement)

1. **`daemon/utils/drift_monitor.py`** — `_CORPUS_DIR` and `BASELINE_PATH`
   were hardcoded relative to the process's CWD instead of the module's own
   file location, silently breaking baseline build/check whenever launched
   from anywhere but the repo root (this is also what caused 2 of the
   originally-reported 8 test failures).
2. **`daemon/router.py`** (`_audit_scan`) — audit records now persist
   `contextify_score` / `sentinel_score` / `shield_score` alongside the
   composite score, so the concept-drift monitor can actually read
   per-pillar distributions from real traffic (previously structurally
   impossible — see correction #17). Confirmed live: after this fix, a
   health check against real scan traffic returned `status: "alert"`,
   `overall_kl: ~2.1–2.8` (varies by run), since the real score
   distribution differs from the synthetic seed baseline — the feature now
   genuinely functions for the first time.
3. **`daemon/pillars/sentinel.py`** (two fixes):
   - A typosquat name that doesn't exist yet on the registry now scores
     100 (was 95), tying rather than trailing an existing typosquat's
     score, per correction #9.
   - `check_registry_existence` now does one confirmatory re-fetch before
     concluding a package doesn't exist, mitigating (not fully solving —
     see Table 2 discussion above) the registry-failure-vs-404 conflation
     that force-BLOCKed `redux-thunk` and `nodemailer` in an early run.
4. **`daemon/pillars/aggregator.py`** — added a force-WARN floor for
   `typosquat_detected`: Sentinel's 0.35 weight alone contributes at most
   35 points, below the 40-point WARN threshold, so a detected typosquat
   with a quiet Contextify/Shield signal (e.g. no real project_path)
   previously sailed through as ALLOW. This alone raised measured typosquat
   recall from 0.37 to 0.75 in the live evaluation. Mirrored the change
   into `daemon/eval/ablation.py`'s local recomputation logic, which was
   already out of sync with the real aggregator in a few other ways
   (missing the covert-dropper amplification and known-incident force-block
   rules, and incorrectly gating the not-found force-block on
   `ai_suggested` when the real aggregator does not) — all now fixed to
   match `aggregator.py`.
5. **`daemon/tests/test_performance.py`** — added an autouse fixture
   mocking npm-registry calls; the six originally-failing latency tests
   were making *real* HTTPS requests to `registry.npmjs.org` on every
   simulated scan (via the disk-footprint check, which none of the test's
   fixtures mocked), which is what blew the millisecond-scale budgets, not
   a routing regression.
6. **`daemon/utils/offline_cache.py`** (`record_allow`) — a transient
   Windows `PermissionError` during concurrent cache-file writes could
   propagate out of `/scan` as an unhandled 500, contradicting the
   function's own documented "never raises" contract. This was directly
   responsible for several of the errors in the first live evaluation run.
7. **`daemon/utils/npm_registry.py`** (three fixes):
   - Added a short-TTL (10s), single-flight cache around the
     full-registry-document fetch. A single `/scan` of an existing package
     independently re-fetched the same package's registry document from up
     to five call sites (Contextify, Sentinel, Shield, disk_checker, and
     the diff analyzer's version-history lookup) with zero sharing between
     them — confirmed via daemon logs showing 9+ identical GET requests for
     one package within a single scan; this was the dominant contributor to
     the tens-of-seconds-per-scan latency in the first live evaluation run
     (38.2s median / 80.6s P95 before the fix).
   - `_get()` only caught `TimeoutException`/`NetworkError`; a dropped
     connection (`httpx.RemoteProtocolError`, a `TransportError` subclass
     not covered by `NetworkError`) propagated unhandled and 500'd the
     scan under real concurrent load. Broadened to catch `TransportError`.
   - The single-flight cache no longer caches a failed fetch (only genuine
     successes) — previously a transient failure was cached for the full
     10s TTL, meaning every other pillar's call for the same package within
     that window inherited the same false "not found."
8. **`daemon/tests/test_sentinel.py`** — one test
   (`test_human_typed_package_skips_hallucination_check`) deliberately made
   a real, unmocked network call to `registry.npmjs.org` "to prove" no
   network call happens for human-typed installs — but the code actually
   does check registry existence unconditionally (only the *hallucination*
   analysis is skipped), so the test's premise was already wrong; it just
   depended on that live call succeeding. Fixed to mock consistently with
   every other test in the file and assert on `check_osv.assert_not_called()`
   instead.

All 449 daemon tests and 99 extension tests pass after these fixes
(`daemon/eval/results/latest.json` and the ablation output were generated
*after* applying them).

## Remaining known limitations (flagged, not fixed, in this pass)

- **Typosquat prefix/suffix coverage gap** and **short-name false positives**
  (`vue`/`vite`, `next`/`nuxt`) — see Table 2 discussion above. Fixing
  requires a redesigned name-similarity check (affix-aware, length-relative
  threshold) validated against a held-out corpus to avoid trading recall
  for new false positives — out of scope for a same-session fix.
- **Registry-failure vs. confirmed-404 conflation** — mitigated (evict
  failed fetches from cache; one confirmatory retry in Sentinel) but not
  structurally fixed. A complete fix needs a tri-state result
  (exists / confirmed-absent / undetermined) threaded from `_get()` through
  every caller of `get_package_metadata`, several of which currently treat
  `None` as a safe default (Contextify, Shield, disk_checker, transitive
  resolution, diff analysis) — auditing all of them safely is a larger,
  separate change.
- **Maintainer-count signal collected but unused** (correction #13) — not
  addressed; flagged as future work in the paper text.
- **Sparse-project 50% weight reduction** (correction #6) — does not
  exist; recommended as a paper correction (remove the claim) rather than
  new code, since the threshold isn't empirically validated.
